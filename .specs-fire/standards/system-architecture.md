# System Architecture

## Overview

`adl` is a terminal-native download manager. The TUI is the only user interface — no GUI windows, menu bar icons, or browser. aria2c runs as a local RPC daemon only while the TUI session is active.

## System Context

### Users

- **macOS power user**: Downloads files, torrents, and magnets from the terminal
- **Homebrew user**: Has aria2c installed via `brew install aria2`

### External Systems

- **aria2c**: Download engine, controlled via JSON-RPC on localhost
- **Homebrew**: Distribution channel for aria2c (dependency) and adl (future)

## Architecture Pattern

**Pattern**: CLI + TUI wrapper around external process
**Rationale**: Leverage aria2c's mature download capabilities; focus adl on UX

## Component Architecture

### Components

#### CLI Layer (`cmd/adl`)
- **Purpose**: Command routing via Cobra
- **Commands**: `adl`, `adl download`, `adl list`, `adl settings`
- **Responsibilities**: Parse args, launch TUI or direct to view, `--help`

#### TUI Layer (`internal/tui`)
- **Purpose**: Interactive Bubble Tea application
- **Views**: Main menu, New Download, Active Downloads, Settings, Help
- **Responsibilities**: User input, progress rendering, keybindings

#### aria2 Manager (`internal/aria2`)
- **Purpose**: Daemon lifecycle and RPC communication
- **Responsibilities**: Start/stop aria2c on TUI open/close, poll download status, pause/resume/cancel

#### Config (`internal/config`)
- **Purpose**: Persistent settings
- **Location**: `~/.config/adl/config.yaml`

#### Download Detector (`internal/download`)
- **Purpose**: Auto-detect input type (HTTP URL, magnet, .torrent path)

## Data Flow

```
User → adl TUI → RPC → aria2c daemon → Network/Disk
                      ↓
              Poll status every ~500ms
                      ↓
              Render progress bars in terminal
```

## Technology Stack

| Layer | Technology | Purpose |
|-------|------------|---------|
| CLI | Cobra | Commands and help |
| TUI | Bubble Tea + Bubbles + Lip Gloss | Interactive terminal UI |
| Backend | aria2c RPC | Downloads |
| Config | YAML | User preferences |

## Non-Functional Requirements

### Performance

- **UI refresh**: ~2 FPS for progress updates (500ms poll)
- **Startup**: Check aria2c within 1 second

### Security

- RPC bound to localhost only
- No background daemon after TUI exit
- No secrets in config file

## Constraints

- macOS primary target (aria2c via brew)
- English UI only
- aria2c-only backend for v1 (no yt-dlp)
- No persistent background process after exit
- Local `go build` for v1; Homebrew formula later

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| TUI style | Mole-like interactive menu | User preference |
| Backend | aria2c RPC (session-scoped) | Progress bars + multi-download while open |
| Daemon lifecycle | Stop on TUI exit | No hidden background process |
| Download location | Default ~/Downloads + folder browser | Fast + flexible |
| Input types | HTTP, magnet, .torrent | aria2c strengths |
| Command name | `adl` | Short, unique, not hijacking aria2c |
| Help | `--help` + in-TUI Help screen | Cobra free + Mole-style tips |
| Install | Local build first | Fast iteration; brew later |
