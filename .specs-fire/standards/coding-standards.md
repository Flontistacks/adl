# Coding Standards

## Overview

Go CLI/TUI project. Keep diffs minimal, match existing patterns, prefer clarity over cleverness.

## Code Formatting

**Tool**: gofmt + goimports
**Enforcement**: Run before commit

## Linting

**Tool**: golangci-lint (when configured)
**Strictness**: Standard Go conventions

## Naming Conventions

| Element | Convention | Example |
|---------|------------|---------|
| Packages | lowercase, single word | `aria2`, `config`, `tui` |
| Exported types | PascalCase | `DownloadManager` |
| Functions | camelCase (unexported), PascalCase (exported) | `startDaemon`, `NewClient` |
| Files | snake_case or single word | `download_list.go` |

## File Organization

```
cmd/adl/          # CLI entry point (Cobra commands)
internal/
  aria2/          # aria2c RPC client and daemon lifecycle
  config/         # Config load/save (~/.config/adl/config.yaml)
  tui/            # Bubble Tea models and views
    views/        # Main menu, download, list, settings, help
  download/       # Download type detection (URL, magnet, torrent)
```

## Error Handling

**Approach**: Return errors up the stack; wrap with context using `fmt.Errorf("...: %w", err)`

**Guidelines**:
- User-facing errors: clear English messages
- Check aria2c presence on startup with `brew install aria2` hint
- Clean shutdown: stop aria2c daemon when TUI exits

## Logging

**Tool**: Standard library `log/slog` or minimal stderr logging
**Format**: Plain text for CLI

**Never log**:
- Full download URLs with credentials
- RPC secrets

## Comments

- Comment non-obvious business logic only
- Bubble Tea `Update`/`View` patterns should be self-explanatory
